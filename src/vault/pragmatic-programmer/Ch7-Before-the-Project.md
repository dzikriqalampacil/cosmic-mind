# Chapter 7 — Before the Project

Part of [[The Pragmatic Programmer]].

## 36. The Requirements Pit

*Perfection is achieved, not when there is nothing left to add, but when there is nothing left to take away…*

**Tip 51: Don't Gather Requirements — Dig for Them**

Requirements are not architecture. Requirements are not design, nor are they the user interface. Requirements are **need**.

**Tip 52: Work with a User to Think Like a User**

**Tip 53: Abstractions Live Longer than Details**

### Manage Requirements Growth

The key to managing growth of requirements is to point out each new feature's impact on the schedule to the project sponsors.

### Maintain a Glossary

It's very hard to succeed on a project where the users and developers refer to the same thing by different names, or refer to different things by the same name.

**Tip 54: Use a Project Glossary**

## 37. Solving Impossible Puzzles

The key to solving puzzles is both to recognize the constraints placed on you and to recognize the degrees of freedom you do have.

**Tip 55: Don't Think Outside the Box — Find the Box**

### There Must Be an Easier Way!

If you cannot find the solution, step back and ask:
- Is there an easier way?
- Are you trying to solve the right problem?
- Why is this thing a problem?
- What is it that's making it so hard to solve?
- Does it have to be done this way?
- Does it have to be done at all?

## 38. Not Until You're Ready

If you sit down to start typing and there's some nagging doubt in your mind, heed it.

**Tip 56: Listen to Nagging Doubts — Start When You're Ready**

Start prototyping to work through the doubt. Choose an area that you feel will be difficult and begin producing some kind of proof of concept.

## 39. The Specification Trap

You should know when to stop:
- A specification will never capture every detail of a system
- The expressive power of language itself might not be enough
- A design that leaves the coder no room for interpretation robs the effort of skill and art

**Tip 57: Some Things Are Better Done than Described**

## 40. Circles and Arrows

**Tip 58: Don't Be a Slave to Formal Methods**

Formal methods have shortcomings:
- Diagrams are meaningless to end users — show a prototype
- Formal methods encourage over-specialization
- Most don't allow dynamic, metadata-driven systems

**Tip 59: Expensive Tools Do Not Produce Better Designs**

Absolutely use formal methods — but remember it is just one more tool in the toolbox.
